# Deploying `@[to_dual]` in Mathlib

This skill covers how to systematically add `@[to_dual]` annotations to Mathlib theorems,
replacing hand-written order-dual versions with auto-generated ones.

## Overview

`@[to_dual]` auto-generates the order-dual of a theorem by translating the proof term.
It guesses the dual name by splitting into camelCase components and looking each up in a
name dictionary (in `Mathlib/Tactic/Translate/ToDual.lean`). For example,
`isLUB_pair` → `is` + `LUB` + `pair` → `is` + `GLB` + `pair` → `isGLB_pair`.

## Identifying candidates

### Strong candidates (convert these first)
Look for pairs where the dual theorem's proof explicitly invokes `.dual` or `@... αᵒᵈ`:
```lean
-- The dual is literally defined via the primal — perfect candidate
theorem lt_isGLB_iff (ha : IsGLB s a) : b < a ↔ ∃ c ∈ lowerBounds s, b < c :=
  isLUB_lt_iff ha.dual
```
Add `@[to_dual]` to the primal (`isLUB_lt_iff`) and delete the hand-written dual.

### Moderate candidates
Pairs where both are proved independently but with structurally parallel proofs
(e.g., both by `simp` or `rw` with dual lemma names). These usually work but need
more careful checking.

### Non-candidates
- **Conjunct order matters**: If the dual has `∃ c ∈ s, a ≤ c ∧ c < b` but `@[to_dual]`
  would produce `∃ c ∈ s, c < b ∧ a ≤ c` (swapped `And`), downstream code that
  pattern-matches on the conjunction will break. Check for this by building importers.
- **Interval theorems with argument swapping**: `Icc a b` on `αᵒᵈ` becomes
  `Icc (toDual b) (toDual a)`. These generally can't use `@[to_dual]`.
- **Name dictionary vs registered translations**: Some type class pairs
  (e.g., `IsDirectedOrder`/`IsCodirectedOrder`) already have constant-level
  translations registered via `@[to_dual]` on their instances. Adding their name
  components (e.g., `directed`/`codirected`) to the name dictionary can *conflict*
  with these registered translations, causing existing `@[to_dual]` annotations
  to produce declarations with mismatched names and types.

## Step-by-step workflow

### 1. Survey the file
Read the entire file and list all dual pairs. Categorize each:
- **Strong**: dual proved via `.dual` or `@... αᵒᵈ`
- **Moderate**: independently proved but structurally parallel
- **Skip**: conjunct ordering, interval argument swapping, type class gaps

### 2. Check the name dictionary
Read `Mathlib/Tactic/Translate/ToDual.lean` and verify the dictionary has entries
for all name components you need. Key entries include:
- `top`/`bot`, `sup`/`inf`, `max`/`min`
- `upper`/`lower`, `above`/`below`
- `least`/`greatest`, `lub`/`glb`
- `minimal`/`maximal`, `succ`/`pred`
- `ici`/`iic`, `ioi`/`iio`, `ico`/`ioc`

### 3. Verify name agreement
For each candidate, check that the auto-generated name matches the existing dual.
The name is generated by translating each camelCase component independently.

**Watch out for word-order mismatches:**
- `isLUB_lt_iff` generates `isGLB_lt_iff` (translating `LUB` → `GLB`)
- `lt_isLUB_iff` generates `lt_isGLB_iff` (translating `LUB` → `GLB`)
- But the hand-written duals might use the OPPOSITE naming convention!

If the names don't match, specify the target name explicitly:
```lean
@[to_dual lt_isGLB_iff]
theorem isLUB_lt_iff ...
```

### 4. Handle attributes
`@[simp]`, `@[gcongr]`, etc. are NOT automatically copied to the dual.
Check whether both the primal and dual had the attribute, and use `(attr := ...)`:
```lean
@[to_dual (attr := simp)]
theorem upperBounds_empty ...
```

### 5. Handle docstrings
Docstrings are not translated. If the dual needs one, pass it explicitly.
**The `(attr := ...)` MUST come before the docstring:**
```lean
/-- Original doc. -/
@[to_dual (attr := simp) /-- Dual doc. -/]
theorem OrderTop.bddAbove ...
```

### 6. Respect declaration order
The dual is generated at the annotation point. If theorem B's dual proof
needs theorem A's dual, then A's `@[to_dual]` must appear first in the file.
Work top-to-bottom through the file.

### 7. Build and test
After each batch of changes:
```bash
lake build Mathlib.The.File           # Build the file itself
lake build Mathlib.Direct.Importer1 Mathlib.Direct.Importer2 ...  # Build importers
```

Find direct importers with:
```bash
grep -rl "import Mathlib.The.File" Mathlib/ | head -20
```

**Common build failures:**
- "not type correct" with instance mismatch → type class pair not in dictionary
- Type mismatch in downstream file → conjunct order changed, or signature differs
- "unexpected token" → docstring/attr syntax wrong (attr must come before docstring)

### 8. Debug failures
Use `to_dual?` to see what name and translation `@[to_dual]` produces:
```lean
@[to_dual?]
theorem isLUB_lt_iff ...
```
Or enable tracing:
```lean
set_option trace.translate_detail true
```

## Adding name dictionary entries

If a translation requires a new dictionary entry, add it to `nameDict` in
`Mathlib/Tactic/Translate/ToDual.lean`. **Always add both directions:**
```lean
  ("atom", ["Coatom"]),
  ("coatom", ["Atom"]),
```

**Danger: conflicts with registered translations.** A new dictionary entry affects ALL
existing `@[to_dual]` annotations in the entire library. Some type class pairs already
have constant-level translations registered via `@[to_dual]` on their instances. A
dictionary entry for their name components can interfere, causing those existing
annotations to produce declarations with mismatched names and types. After adding an
entry, build broadly — not just the file you're working on. The `directed`/`codirected`
pair is a known example where adding the entry causes failures in `Order/Directed.lean`.

## Common patterns

### Strongest pattern: dual proved via `.dual`
```lean
-- Before:
theorem isGLB_lt_iff (h : IsGLB s a) : a < b ↔ ... := lt_isLUB_iff h.dual

-- After: delete the above, add @[to_dual] to the primal
@[to_dual isGLB_lt_iff]
theorem lt_isLUB_iff (h : IsLUB s a) : b < a ↔ ... := ...
```

### Pattern: both `@[simp]`, dual proved via `@... αᵒᵈ`
```lean
-- Before:
@[simp] theorem upperBounds_empty : ... := by simp [...]
@[simp] theorem lowerBounds_empty : ... := @upperBounds_empty αᵒᵈ _

-- After:
@[to_dual (attr := simp)]
theorem upperBounds_empty : ... := by simp [...]
```

### Pattern: dual with docstring
```lean
-- Before:
/-- Global max bounds above. -/
@[simp] theorem OrderTop.bddAbove ... := ...
/-- Global min bounds below. -/
@[simp] theorem OrderBot.bddBelow ... := @OrderTop.bddAbove ...

-- After:
/-- Global max bounds above. -/
@[to_dual (attr := simp) /-- Global min bounds below. -/]
theorem OrderTop.bddAbove ... := ...
```
